# Model Hypothesis

## Purpose

This document captures the current model hypothesis for the target domain.

Use it during `extract`, `refine`, and `build` to define vocabulary, boundaries, use cases, and slice candidates.

---

## Current Hypothesis

`easy-login` is a minimal identity service for browser-first games and simple applications.
Its primary promise is continuity without forced login.
Users should be able to enter quickly, keep a persistent identity on the same browser, and optionally upgrade that identity into something portable and recoverable.

The system should optimize for:

- instant access
- persistent identity without mandatory full authentication
- progressive ownership instead of upfront account creation
- recovery after device loss when the user has claimed the identity

## Core Concepts

- `Player`: the persistent identity record used by the system
- `DisplayName`: the human-visible name shown to other users
- `PlayerId`: the internal stable identifier tied to ranking and continuity
- `DeviceToken`: a browser-local token that links a returning browser to a player identity
- `Claim`: the act of attaching ownership proof to an existing player identity
- `RecoveryPassphrase`: the proof used to reclaim a claimed identity on another device

## Likely Entities And Value Objects

### Likely entities

- `Player`
- `ClaimedIdentity` or equivalent ownership state attached to a player
- `DeviceRegistration` or equivalent binding between browser state and player state

### Likely value objects

- `DisplayName`
- `PlayerId`
- `DeviceToken`
- `RecoveryPassphrase`

## Major State Transitions

- anonymous visitor chooses a display name and receives a persistent guest identity
- returning browser presents a valid device token and resumes the same player identity
- guest identity is claimed through a recovery passphrase
- claimed identity is recovered on another device using the recovery passphrase
- display identity may change while persistent identity remains stable

## Boundary Assumptions

- display identity is not the same thing as persistent identity
- ownership proof is separate from normal session continuity
- ranking or other continuity-sensitive features should attach to `PlayerId`, not to display name
- no email, OAuth, or heavyweight login should be required in the initial model
- the system should tolerate storage loss on an unclaimed identity but should offer recovery after claim

## Candidate Use Cases

- create guest identity from display name
- resume identity from device token
- claim identity with recovery passphrase
- recover claimed identity on a different device
- update display name without losing ownership or continuity
- read current identity status for a client session

## Expected Architecture Direction

The initial implementation direction should use the `contacts` repository as an architecture reference, not as a domain template.

Expected reference traits:

- monorepo with separate frontend and backend applications
- Mithril-based frontend application
- Go backend application
- explicit contract-sharing area for API descriptions or request artifacts when needed
- separate container build targets for frontend and backend

Transferable patterns observed in `contacts`:

- `apps/web` and `apps/api` separation
- simple frontend route composition around Mithril
- backend HTTP service with explicit health and readiness endpoints
- container-first local and deployment posture

These are directional expectations for refinement and build, not final implementation decisions.

## Unresolved Tensions

- whether display names are unique only after claim or remain permanently non-unique
- whether claim is generated by the system, user-supplied, or supports both modes by default
- whether one player can register multiple active devices and how that affects recovery
- how much abuse prevention is needed before introducing heavier identity controls
- what persistence technology best fits the initial slice while preserving later portability
